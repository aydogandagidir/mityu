# Architecture Decision Records (ADR)

Append a short ADR whenever you make a decision that shapes structure, dependencies, security, or the tenancy/local-first model. Format: Context → Decision → Consequences → Status.

## ADR-0001 — Build on the Tauri/Rust core, treat the Python backend as legacy
**Context:** Meetily's supported app is the Tauri (Rust) + Next.js client; the Python/FastAPI `backend/` is archived with unauthenticated CORS.
**Decision:** Build the product on the Tauri core. Do not depend on the legacy backend at runtime; use it as reference (schema/prompts) only. The future server is a NEW clean service.
**Consequences:** One authoritative capture/transcribe/summarize path; a clean, secure server later; some reference code stays unused.
**Status:** Accepted.

## ADR-0002 — Local-first, server-optional, tenant-aware-by-design
**Context:** We want enterprise + eventual multi-tenant SaaS without losing the local-first (privacy) value prop.
**Decision:** Ship single-tenant/local first; introduce the four seams (AuthContext, tenant_id on all entities, repository layer, dormant sync) from Phase 1. Server is additive.
**Consequences:** Phase 2 is additive, not a rewrite; a little upfront discipline (tenant_id, repositories) in Phase 1.
**Status:** Accepted.

## ADR-0003 — Server language: Rust/Axum
**Context:** Options: Rust/Axum (cohesion with the core, shared models) vs clean FastAPI (velocity, reuse legacy schema/prompts as reference). Decided 2026-07-02 at BOOTSTRAP Step 3, per the default recommendation in docs/ARCHITECTURE.md.
**Decision:** **Rust/Axum** for the Phase-2 sync/collaboration server. Rationale: one language across core+server (the docs/CONTRACTS.md types — AuthContext, Repository, sync protocol — map 1:1 to shared Rust types instead of being re-modeled in Python), one toolchain (already proven on this repo), no second runtime to secure/operate, and the archived FastAPI backend stays reference-only for schema/prompts. FastAPI remains the sanctioned fallback ONLY if Rust server iteration proves measurably slow.
**Consequences:** `server/` (Phase 2, task D1) scaffolds as Axum + Postgres (RLS) with OIDC at the gateway; shared entity types live in a workspace crate reusable by client and server; hiring/onboarding assumes Rust.
**Status:** Accepted (re-confirm briefly at D1 kickoff before the first `server/` commit; server work does not start before gate C8).

**Context:** Base has both `audio/` and `audio_v2/`. Read-only investigation (audio-pipeline-engineer, 2026-07-02, Meetily v0.4.0) gathered wiring/runtime/history evidence.
**Decision:** **`frontend/src-tauri/src/audio/` is the authoritative module.** `audio_v2/` is an abandoned v0.1.1 rewrite scaffold, **not compiled into the binary**: no `mod audio_v2;` anywhere (`lib.rs:40` declares only `pub mod audio;`), zero `crate::audio_v2::` references (vs **59** refs to `crate::audio::` across 21 files), zero `#[tauri::command]` entry points (all ~30 registered audio commands come from `audio::`), and it is bit-rotted (imports `crate::audio::core::*` which no longer exists; duplicate `MixingMode`; undefined `AudioStream`). Git: `audio/` 125 commits (active through v0.4.0); `audio_v2/` exactly 1 commit (2025-10-23, v0.1.1) then never touched; 19 TODO/"Phase N" stubs in 1,393 LOC. Live-session log targets (`app_lib::audio::pipeline`, `::vad`, `::transcription::worker`) confirm `audio/` is the executing pipeline. The improvements audio_v2 promised (simplified pipeline, EBU R128) were since implemented inside `audio/` itself (`audio/mod.rs:18-26`, `audio_processing.rs`).
**Consequences:** No convergence flag is needed — nothing reachable to converge. `audio_v2/` is FROZEN (no new code, no imports). Delete it in a dedicated audio-cleanup ticket **after the Phase-0 validation gate passes**, together with the other verified-unreferenced dead files (`src/lib_old_complex.rs`, `src/audio/core-old.rs`, `src/audio/recording_saver_old.rs`), followed by `cargo build` + one recording smoke test per platform. All audio work targets `audio/` and routes to audio-pipeline-engineer.
**Status:** Accepted.

## ADR-0010 — Tenant-scoped repositories are the only storage access path (BACKLOG B2 phase 2)
**Context:** Phase 1 (migration `20260702000000`) added `workspace_id` + sync columns to every domain table; B1 added the `AuthContext` seam. The repositories still ran unscoped SQL, and stray SQL existed outside `database/` (`api/api.rs` `open_meeting_folder`; `audio/import.rs` meeting+segment inserts and settings read; `audio/retranscription.rs` transcript delete/insert and two settings reads).
**Decision:** Every public method of the five repositories (`meeting`, `transcript`, `summary`, `transcript_chunk`, `setting`) takes `ctx: &AuthContext`; every SELECT/UPDATE/DELETE is scoped by `workspace_id = ctx.tenant_id`; every INSERT sets `workspace_id` and populates `created_at`/`updated_at` (nullable on `transcripts`/`transcript_chunks`/settings tables); every write to a synced table bumps `rev` and stamps `updated_by`. Commands and background tasks resolve identity **only** via `crate::context::current()` (never from the frontend — `AuthContext` is deliberately not `Deserialize`). Stray SQL was routed through repository methods; two methods were added for the audio flows (`create_meeting_with_segments` — preserves caller segment ids; `replace_meeting_transcripts` — transactional delete+insert with a scoped meeting-existence check) and one projection (`get_transcript_provider_model`). Timestamp writers standardized on binding `chrono::DateTime<Utc>` (RFC 3339); the one `naive_utc()` outlier (`update_meeting_title`) was fixed. Reads do **not** filter `deleted_at` yet — the app has only hard-delete flows today; soft-delete semantics land with sync.
**Settings caveat:** `settings`/`transcript_settings` keep the legacy single-row `id = '1'` PRIMARY KEY, which collides across workspaces. Until a Phase-2 migration widens the key to `(id, workspace_id)`, their upserts guard `DO UPDATE ... WHERE workspace_id = excluded.workspace_id`, so a foreign workspace's write degrades to a no-op instead of clobbering another workspace's row (proven by test). Single-workspace local mode is unaffected.
**Consequences:** `grep` for `sqlx::query` outside `database/` is now a valid review gate (`/tenant-check`); local flows are behavior-identical (everything is workspace `local`); cross-workspace isolation is proven by `frontend/src-tauri/tests/repository_tenant_scoping.rs` (6 integration tests, both directions). `rev` now increments on local edits — correct "modified since baseline" state for future sync.
**Phase-2 rules (guardian review, 2026-07-02):** (a) when `context::current()` becomes request-scoped, spawned background tasks must receive the `AuthContext` **captured at command time** — they must not re-resolve `current()` inside the task (today's constant resolver makes both equivalent; the spawn site in `summary/commands.rs` carries a `// Phase 2:` marker). (b) Repository no-match results (`Ok(false)`/0-rows on cross-workspace access) map to **404** at Phase-2 server endpoints, and status-writer methods (`update_process_completed/failed/cancelled`) must check `rows_affected` server-side rather than returning unconditional success.
**Status:** Accepted.

## ADR-0011 — BYOK LLM/STT API keys live in the OS credential store, not SQLite plaintext
**Context:** CLAUDE.md §0.7/§3 and `docs/SECURITY_PRIVACY.md` ("Secrets") require BYOK keys to live in the OS keychain / secure store — never SQLite plaintext, source, or logs. The Meetily base stored provider keys in plaintext `*ApiKey` columns of `settings` / `transcript_settings` (flagged by the guardian security review). This is a standalone secrets fix; **it is not** the SQLCipher at-rest DB-encryption work (BACKLOG B3, deferred — that will be a later ADR and encrypts the whole DB file; the two are independent and complementary).
**Decision:** Introduce a `secrets` module (`frontend/src-tauri/src/secrets.rs`, registered in `lib.rs`) wrapping the `keyring` crate (v3.6.3; native backends only — Windows Credential Manager via `windows-native`/`windows-sys` pure-winapi, macOS Keychain via `apple-native`, Linux keyutils via `linux-native`; no dbus/openssl build-time toolchain, works fully offline). Service name = `com.bluedev.mityu`; entry name = `{tenant_id}:{summary|transcript}:{provider}:api_key` — **workspace-scoped** (tenant-aware-by-design; a future non-`local` workspace cannot read another workspace's key, identity from `AuthContext` only) **and domain-scoped** (so the two settings tables' shared provider names `openai`/`groq` do not collide). All secret functions take `&AuthContext`, return `anyhow::Result` with context, **never log key values**, and **fail closed** — a locked/unavailable store yields a user-facing error with **no silent plaintext fallback**.
`SettingsRepository::save_api_key`/`save_transcript_api_key` now write the secret to the credential store first (abort before touching the DB if it fails), then persist only the non-secret marker `keychain:v1` in the legacy column; `get_api_key`/`get_transcript_api_key` read from the credential store; `delete_api_key` clears both. The columns are **kept** (not dropped) for schema/older-binary compat and to carry the reference marker. The existing Tauri command signatures/return types are unchanged, so the Next.js UI keeps working through the same `invoke()` calls with no `frontend/src` edits.
A one-time migration `migrate_plaintext_keys_to_keychain` runs inside `DatabaseManager::new` (covers fresh/legacy-import/recovery init paths) and lazily on first read: for any column still holding a real key (present, non-empty, not the marker) it moves the value into the credential store and overwrites the column with the marker. It is **idempotent**, iterates **every workspace row** (uses each row's own `workspace_id`), runs **offline**, and is **non-fatal** — a keychain failure at startup logs a warning and is retried lazily, so the app still opens the DB and works offline (local-first invariant).
**Consequences:** No provider key is ever persisted in SQLite again; local single-`local`-workspace behavior is byte-identical for the user (a previously-saved key still calls providers after migration). The `geminiApiKey` column (schema-present but not yet wired to a provider) is swept defensively. `keyring` is now a dependency (and dev-dependency, so integration tests can install a persistent in-memory credential store — keyring's built-in `mock` is `EntryOnly` and cannot model separate `Entry::new` set→get calls). Tests never touch the real machine credential store. **For the SQLCipher (B3) agent:** `keyring 3.6.3` is already in `Cargo.toml`; B3 encrypts the DB file at rest and is orthogonal to this — the `*ApiKey` columns will by then hold only the marker, so no plaintext secret is exposed even in the pre-encryption DB.
**Status:** Accepted.

## ADR-0012 — Dormant sync client seam (BACKLOG B4): typed, disabled, no network, apply-remote bypasses the Phase-1 repositories
**Context:** ADR-0002 mandates a **dormant sync client** as one of the four Phase-1 seams so Phase 2 (the optional `server/`, ADR-0003 Rust/Axum) is additive, not a rewrite. `docs/CONTRACTS.md` §5 pins the wire shapes; `docs/DATA_MODEL.md` defines the sync semantics (monotonic `rev`, `updated_by`, soft-delete `deleted_at`; client SQLite authoritative for a user's own captures) and the never-synced baseline (`rev = 1` + `updated_by IS NULL`); `docs/MULTITENANCY.md` classifies data as local-only vs synced (`provider_credential` secrets NEVER sync). The seam must exist and be typed **without** breaking the local-first invariant: the app must behave byte-identically with sync off, and **no** network dependency may be added (no `reqwest`/`tokio-tungstenite`).
**Decision:** Add `frontend/src-tauri/src/sync/` (`mod.rs` / `protocol.rs` / `client.rs`, per `docs/SCAFFOLD.md`), registered as `pub mod sync;` in `lib.rs`. It is **inert**: no Tauri command is registered, nothing on the capture→transcript→summary→store path calls it, and there is no UI toggle. Reuses only already-present deps (serde, serde_json, uuid, chrono, thiserror, anyhow).
- **`protocol.rs`** — the §5 wire types with serde: `SyncEntity { Meeting, Transcript, Summary, ActionItem }` (`#[serde(rename_all = "snake_case")]` ⇒ tokens `meeting|transcript|summary|action_item`), `PushItem { tenant_id, entity, id, rev: u64, updated_by: Option<String>, updated_at, deleted: bool, payload: serde_json::Value }`, and `ServerAck { id, server_rev: u64, conflict: bool }`. Struct field order matches §5 so the serialized key order matches the documented sample; round-trip/shape tests assert the exact key SET, ORDER, and value types against JSON literals transcribed from §5, and parse the §5 doc samples back. `updated_by` is `Option` so the never-synced baseline serializes as `updated_by: null` (no sentinel user id).
- **`client.rs`** — `SyncConfig { enabled: bool }` (`Default` derived ⇒ `false`), a `SyncClient<T: Transport = NoopTransport>` whose `push`/`pull` **guard on `enabled` FIRST and return `SyncError::Disabled` (thiserror) performing NO I/O** while disabled (proven by an atomic-flag spy transport that stays untouched). A `Transport` trait abstracts the wire; the ONLY Phase-1 impl is `NoopTransport` (reports non-operational, returns `SyncError::NoTransport` — never fabricates acks), so the Phase-2 HTTPS transport slots in as a new impl behind a flag without changing this file. `SyncError` also pre-defines `NoTransport`/`Transport(String)`/`Protocol` for a stable Phase-2 surface.
- **CRITICAL seam — `RemoteApply` (guardian-flagged):** applying a PULLED remote change MUST NOT go through the Phase-1 repositories (ADR-0010), because those bump `rev = rev + 1` and stamp `updated_by = ctx.user_id` on **every** write. Replaying a remote change through them would (1) masquerade it as a local edit (`updated_by` = local user), (2) re-bump `rev` and destroy the server-assigned `rev` and the `rev = 1 / updated_by NULL` never-synced baseline, and (3) make the row look "modified since sync" again → the next push re-sends it, the server re-acks, forever (a ping-pong / infinite echo). So `apply_remote` is a **distinct trait** (defined, not wired) whose contract — captured in rustdoc — is to write `rev`/`updated_by`/`updated_at`/`deleted_at` **VERBATIM** from the `PushItem` payload (a future dedicated "remote-apply" repository path with its own `... ON CONFLICT DO UPDATE SET rev = :remote_rev, updated_by = :remote_updated_by, ...` SQL), passing `AuthContext` for **tenant scoping only**, never to derive identity/rev. A unit test encodes this as an executable check (a recording `RemoteApply` impl preserves a remote author + rev, distinct from `ctx.user_id`).
- **Classification (local-only vs synced):** `SyncEntity` enumerates only the four `Synced? = yes` domain entities. There is deliberately **no** `Settings`/`Credential` variant, so a `PushItem` for `settings`/`transcript_settings` or a `provider_credential` secret cannot even be constructed — the local-only/secret boundary is structural, not disciplinary. Only a non-secret label may ever sync (a future, separate decision).
**Consequences:** The seam is typed and reviewable; Phase 2 fills in the `push`/`pull` bodies, adds an HTTPS `Transport`, and implements `RemoteApply` as a separate repository path (never a method on the existing `database/repositories/` types). Local-first holds: with sync off (always, today) behavior is byte-identical and offline; no network crate was added. DoD verified: `cargo build -p mityu` green; `cargo clippy -p mityu --all-targets` unchanged at the 142-warning baseline (the one initial `derivable_impls` finding on `SyncConfig::Default` was resolved by deriving `Default`); `cargo test -p mityu` passes (12 new sync tests + existing suites); sync files `rustfmt`-clean.
**Status:** Accepted (re-confirm the wire shapes against `server/` at D1 before the first sync endpoint).

## ADR-0013 — On-device AI agents: local-first, draft-only (HITL), no autonomous actions (EPIC F seam)
**Context:** The About screen advertises "Coming soon: a library of on-device AI agents — automating follow-ups, action tracking." That was marketing ahead of the plan: BACKLOG had **source-linked summaries (C1)** and **action-item extraction (C2)** but no "agents" epic and no "automated follow-up" capability. Building agents naively (autonomous send, cloud round-trips, auto-approval) would violate `CLAUDE.md` §0.1 (local-first), §0.5 (HITL for every AI output), and §10 (no autonomous irreversible actions). Separately, the About "Works everywhere — Google Meet / Zoom / Teams" is **not** an API integration and is intentionally absent from the plan: the app captures **system audio** (WASAPI loopback / BlackHole), recording any call online or offline — `CLAUDE.md` §1 "not a meeting bot that joins calls" stands.
**Decision:** Introduce **BACKLOG EPIC F** and scaffold a **dormant `agents/` seam** now (`frontend/src-tauri/src/agents/` = `mod.rs` + `draft.rs` + `engine.rs`, `pub mod agents;`), following the ADR-0012 dormant-`sync/` precedent (typed, disabled, no network dependency, tests pin the invariants). Reuses only already-present deps (serde, serde_json, thiserror). Four rules, encoded structurally (not merely documented):
1. **Local-first / dormant:** no LLM/HTTP client in the seam; Phase-F agents reuse the existing summary providers (`summary/` + anthropic/openai/groq/ollama/openrouter, BYOK, offline) behind the `Agent` trait (Phase-1 impl = `NoopAgent`, no I/O). Off by default (`AgentsConfig { enabled: false }`, `Default` derived) ⇒ a build is byte-identical to one without the module.
2. **HITL:** an agent run yields an `AgentDraft` whose status `AgentRunner::run` forces to `Draft`; approval is a separate human action. A self-approving agent cannot bypass it (pinned by `run_forces_hitl_never_returns_approved`). `run` guards on `enabled` FIRST and returns `AgentError::Disabled` doing no work (an atomic-flag spy proves the agent is never touched — the same pattern as sync).
3. **No autonomous external actions:** agents only *propose* — `ProposedAction` is data; there is **no** execute/send method anywhere in the seam. "Send"/"create" stays a manual user step (`CLAUDE.md` §10).
4. **Source-linked + tenant-scoped:** every output carries `SourceRef` (→ `source_chunk_id`, C1) and `ctx: &AuthContext`; `AgentRun` carries the §6 sync-ready fields (`id`, `tenant_id`, `created_at`/`updated_at`, `updated_by: Option`, `rev`, soft-delete) so a Phase-2 `agent_runs` table + sync are additive — with no `SyncEntity` coupling yet (agents don't sync until a separate decision).
Sequenced **after gate C8**, dependent on **C1 + C2**; trigger model manual/on-demand first, scheduling only at F5 behind explicit opt-in.
**Consequences:** The "Coming soon" copy is now backed by a tracked, gated plan (BACKLOG F0–F5) and an inert, tested seam that ships today without changing behavior or adding a runtime dependency. Verified here with `rustfmt` (all three files parse clean & formatted); the full `cargo build -p mityu` / `clippy` / `test` gates run in an environment with the LLVM/CMake toolchain (BOOTSTRAP Step 1). The seam's `#[cfg(test)]` invariant tests mirror `sync/`'s and only compile under `cargo test`, so a test issue cannot break `tauri:dev`. Revisit trigger/automation policy at F5 before any scheduled run ships.
**Status:** Accepted (seam merged dormant; F0 re-confirms scope at C8 kickoff).

## ADR-0014 — Encryption at rest (SQLCipher): key in OS keychain, one-time plaintext→encrypted conversion
**Context:** `docs/SECURITY_PRIVACY.md` ("Encryption — At rest (client): SQLCipher (encrypted SQLite) for sensitive data; OS-protected key") and BACKLOG B3 require the local SQLite DB to be encrypted at rest with an OS-protected key, while the app must **still open existing data** (documented migration path) and stay **local-first** (works fully offline; a key/store failure must never silently downgrade to plaintext). ADR-0011 already moved BYOK provider secrets out of SQLite into the OS keychain and explicitly deferred whole-file DB encryption to B3 (the two are orthogonal and complementary — after B3 the `*ApiKey` columns still hold only the `keychain:v1` marker, so even the pre-encryption backup exposes no provider secret). Feasibility was proven in a prior Step-1 spike: `sqlx 0.8.6` → `sqlx-sqlite` pulls `libsqlite3-sys 0.30.1`, which ships a `bundled-sqlcipher-vendored-openssl` feature, and sqlx treats the SQLCipher `key` pragma as a reserved first-executed slot.
**Decision:** Encrypt the whole `meeting_minutes.sqlite` file with SQLCipher (AES-256), keyed by a device-scoped 256-bit key held in the OS credential store.
- **Cargo (two dep changes):** (1) declare `libsqlite3-sys = { version = "0.30.1", features = ["bundled-sqlcipher-vendored-openssl"] }` as a **direct** dependency purely to enable the feature — Cargo unifies it with sqlx's transitive `bundled` on the same crate version, so sqlx links a SQLCipher-capable libsqlite3; no system OpenSSL/SQLCipher toolchain is needed (the vendored OpenSSL + SQLCipher amalgamation compile from source, one long native build). (2) declare `zeroize = { version = "1.8", features = ["alloc"] }` (already in-tree transitively at 1.8.2, so it unifies with no new crate) to scrub the DB key hex from memory (security-review hardening, below).
- **Key management (`secrets::db`, reuses ADR-0011's `keyring` 3.6.3):** a new `db` submodule in `secrets.rs` stores/reads the DB key under service `com.bluedev.mityu`, entry `db-key`. Unlike BYOK keys it is **device-scoped, not workspace-scoped** (one encrypted file spans all workspaces). `get_or_create_hex()` generates 32 bytes from the OS CSPRNG (`rand` 0.8, already in-tree) on first launch, stores them as 64 lowercase hex chars, and reads them back on every startup — fully offline. The key value is **never logged**; a malformed stored value is rejected (never silently regenerated — that would strand decryptable data); the value is passed to SQLCipher as a **raw** key `PRAGMA key = "x'<hex>'"` (no KDF). The key hex is returned as `zeroize::Zeroizing<String>` and held that way in `database::manager` (dropped right after the pool opens), so the material is scrubbed from freed heap rather than lingering.
- **Keyed pool (`database/manager.rs`):** the main pool moved from `SqlitePool::connect(path)` to `connect_with(SqliteConnectOptions…)` applying `.pragma("key", x'<hex>')` (sqlx executes the reserved `key` slot first, before any statement), and preserves the prior behavior explicitly with `.journal_mode(Wal)` + `.foreign_keys(true)` + `create_if_missing(true)`, so the existing WAL checkpoint/recovery paths (`cleanup`, orphaned-WAL retry) keep working and a fresh install is created already-encrypted. The key is fetched **fail-closed**: a locked/unavailable keychain maps to a `sqlx::Error::Configuration` and the DB is **not** opened unencrypted.
- **One-time conversion (`database/encryption.rs`, before migrations):** `ensure_encrypted()` runs in `DatabaseManager::new` **before** the keyed open and **before** `sqlx::migrate!`. It detects a plaintext DB by the `"SQLite format 3\0"` header (absent on a cipher file), then: (1) **checkpoints any plaintext `<db>-wal`** (`PRAGMA wal_checkpoint(TRUNCATE)`) so committed WAL-only rows fold into the main file and no stale plaintext `-wal` can collide with the encrypted DB's WAL; (2) opens the **encrypted destination** as the main keyed connection (create) **in rollback-journal (DELETE) mode** so the cipher DB is a single self-contained file (no `<dst>-wal`/`-shm` orphans a single-file rename would leave), `ATTACH DATABASE '<plaintext src>' AS plaintext KEY ''`, `SELECT sqlcipher_export('main','plaintext')`, `DETACH`; (3) an atomic file swap moves the original to a **temporary** `<db>.pre-encryption` backup and promotes the cipher file; (4) **verifies** the promoted file opens with the key (`SELECT count(*) FROM sqlite_master`) — and **only then deletes** the `.pre-encryption` backup (scrub-then-unlink) and any stale plaintext `<db>-wal`/`-shm`, so **no full-database plaintext lingers on disk**. If verification fails, the plaintext backup is restored and startup aborts (fail closed, recoverable). (The reverse direction — encrypted-main + plaintext-attach — is required because this `libsqlite3-sys 0.30.1` build returns `SQLITE_CANTOPEN` when `ATTACH` is asked to *create* a new encrypted file; it also mirrors exactly how the live pool later opens the keyed file, so a successful+verified conversion guarantees the pool opens it.) No row is rewritten, so every `workspace_id`/`rev`/`created_at`/`updated_at` and the **`_sqlx_migrations` ledger survive byte-for-byte** (asserted in tests). It is idempotent (no-op on an already-encrypted or missing file). The legacy `.db`→`.sqlite` import still lands plaintext first and is then encrypted by the same path.
- **Migration path for existing users:** on the next launch after upgrade, an existing plaintext `meeting_minutes.sqlite` is transparently converted once. The `.pre-encryption` backup is a **transient** safety copy during conversion and is **deleted after the encrypted DB is verified readable** — so afterwards no plaintext copy remains. Because the plaintext backup is gone and the key lives only in the OS keychain, deleting/rotating the `db-key` entry or restoring on a new machine without it makes the DB **permanently unrecoverable** — this is the intended at-rest posture (an attacker with the file but not the keychain gets nothing). A user-facing key export/rekey affordance is future work (below); until then, protecting the OS keychain is the recovery story (documented in `docs/SECURITY_PRIVACY.md`).
- **Keyless-still-reads-plaintext invariant:** a SQLCipher build with **no** `key` pragma still opens ordinary plaintext SQLite files, so the existing integration suites (`migration_workspace_sync`, `repository_tenant_scoping`) that open plain temp DBs with no key keep passing unchanged (pinned by a dedicated test).
**Consequences:** The local DB is encrypted at rest with an OS-protected key; local-first holds (offline, no network); no provider secret is exposed at any point (ADR-0011). **After a verified conversion no full-database plaintext remains on disk** (the temporary `.pre-encryption` backup and stale plaintext WAL/SHM are removed), and the DB key hex is zeroized from memory after the pool opens. Cost: one long first native build (vendored OpenSSL + SQLCipher), and the deliberate loss of any plaintext fallback — **the OS keychain `db-key` entry is now the sole recovery path; losing it = unrecoverable data** (correct at-rest posture). Security/guardian review (2026-07-02) passed with no blocker; the three fixes applied here were: delete the plaintext backup after a verified keyed re-open (HIGH), checkpoint the plaintext WAL before conversion + remove stale plaintext WAL/SHM after (MED), and zeroize the key hex (LOW hardening). Follow-ups: a user-facing **key export/rekey** affordance (so a user can back up/migrate their key before the backup is gone), and the sync/backup denylist for the encrypted file + key (deferred to B4). New tests: `frontend/src-tauri/tests/db_encryption_sqlcipher.rs` — fresh encrypted DB + migrations; plaintext→encrypted conversion preserving row counts **and** the `_sqlx_migrations` ledger (seeded via the public `Migrator`) **and** asserting the `.pre-encryption` backup is deleted; a populated-`-wal` conversion that preserves committed rows and leaves no stale plaintext `-wal`/`-shm`; wrong-key-fails-closed; the keyless-plaintext invariant; and an env-gated (`MITYU_TEST_ENCRYPT_DB`) real-copy conversion proof (copy the live DB, convert, assert counts + ledger + backup-deleted + no plaintext residue). Touched only: `Cargo.toml` (the two features), `secrets.rs`, `database/manager.rs`, `database/mod.rs`, new `database/encryption.rs`, the new test, and this ADR + the SECURITY_PRIVACY "Encryption" note.
**Status:** Accepted (SQLCipher at-rest encryption merged; no plaintext residue after conversion; revisit key export/rekey + sync/backup denylist when sync (B4) or a second device lands).
